## 一、四条腿定义与依赖关系

上线 = 四条彼此**大体独立**的腿并行推进,只在两处有硬依赖(见箭头):

| 腿 | 名称 | 性质 | 依赖 |
|----|------|------|------|
| ① | ICP 备案 + 域名 | 行政线(墙钟不可压缩) | 无(最先点火);是 ③ 绑自定义域名的**前置** |
| ② | ADR-004 内容安全 | 工程主菜 | 无(不依赖其余三腿) |
| ③ | CloudBase 部署接缝 | ADR-002 首次真跑 | 排在 ② 之后(密钥管理一次定型);绑自定义域名依赖 ① 备案下号 |
| ④ | 最小可上线形态取舍 | 决策项 | 无(可随时敲定);成本闸门为软启动**硬门槛** |

**依赖关系图**:

```
① ICP 备案 + 域名 ──(备案号)──▶ ③ 绑自定义域名
② 内容安全 ──(审核 API 密钥 + 出网形态)──▶ ③ 密钥管理一次定型
④ 最小形态 ── 独立,但其「成本闸门」是软启动放量的硬门槛
```

- ①②④ 三条可**同时开工**;③ 是收口腿(要等 ② 定型密钥管理、等 ① 下备案号才能绑正式域名)。
- ③ 在 ① 备案号未下来前可先用 CloudBase 默认二级域名跑通,备案号到位再绑自定义域名——即
  ③ 的「部署跑通」不必等 ①,只有「绑自定义域名」等 ①。

## 二、排序结论与理由

**点火顺序:① 立即启动 → ②(备案等待期内做)→ ③(② 后收口)→ ④(随时敲定,软启动前锁死)。**

逐条「为什么不是别的顺序」:

1. **① 最先,因为它是唯一墙钟不可压缩的行政线。** ICP 备案审核周期以「工作日/周」计、由管局排期,不吃工程投入也压不短。越早点火,越能把备案等待期变成 ② 的工程窗口而非纯等待。放到最后 = 备案周期直接顶格延后上线。

2. **② 排在备案等待期,因为它不依赖其余三腿、且是纯工程可自控进度的最大块。** 内容安全是「生成内容必须过审」的合规硬要求(§六雷区),也是四腿里工程量最大的一块。趁 ① 在管局排队、③ 还没到收口时机,② 正好填满这段窗口——把不可压缩的行政等待和可压缩的工程投入**并行**,而不是串行。

3. **③ 排在 ② 之后,因为密钥管理要一次定型。** ③ 首次把 ADR-002(Spring Boot @ CloudBase 云托管)真跑起来,会引入运行时密钥(DeepSeek key + ② 的审核 API key)。若 ③ 先于 ② 定型,② 落地时又要回头改一遍密钥注入/出网配置。排 ② 后 → ③ 一次把「运行模型 key + 审核 key」的密钥管理形态定死,不返工。

4. **④ 可随时敲定、但成本闸门必须在软启动前锁死。** 最小可上线形态是决策项(不吃行政周期、不吃大工程),可以任何时候拍;唯一的硬约束是它的「成本闸门」——软启动一旦放真实用户进来,没有全局熔断就等于把 API 账单敞口交给陌生流量,所以成本闸门是**软启动放量的硬门槛**,必须先于「放人进来」落地。

## 四、① ICP 备案 + 域名 · 行政清单与待确认项

> **性质:Felix 自查、非工程项。** 本节只列清单与待确认,不代查、不代拍。

**行政清单(个体工商户 + `.cn` 域名路径)**:

1. 备妥个体工商户营业执照(备案主体证件)。
2. 域名 `wanjie.cn` 可注册性核查 → 在支持 `.cn` 实名的注册商注册,完成域名实名认证(与备案主体一致)。
3. 走 ICP 备案(经营性 or 非经营性视变现形态定;软启动免费阶段先非经营性,接微信支付变现前再评估经营性/增值电信许可)。
4. 备案通过、下备案号 → 回填给 ③ 绑自定义域名 + 页面底部展示备案号。

**待确认三项(Felix 自查)**:

- **(a) 执照注册地 / 管局**:个体工商户执照的注册地决定归属哪个省级管局,备案规则/周期随管局略有差异。
- **(b) 经营者远程核验可行性**:多数管局要求经营者本人做 App 刷脸/视频核验;确认能否全程远程完成(异地/无法到场时的替代路径)。
- **(c) CloudBase 能否作备案载体**:确认腾讯云 CloudBase 是否提供备案接入(备案通常绑「服务器/云资源」提供商);若 CloudBase 走腾讯云备案系统,则备案与 ③ 的托管商天然同源、省一次迁移。

## 五、② ADR-004 内容安全 · 待答问题清单

> **纪律:以下全是「待答问题」,不在本文预先拍板任何选型结论。** ADR-004 正文待独立工作单元起。
> 倾向仅记为倾向,不等于决定。

**待答问题**:

1. **审核 API 选型** — 候选:腾讯云天御 TMS(文本内容安全)/ 网易易盾 / 阿里云绿网。
   *倾向* TMS(与 ③ CloudBase 同账号同生态、密钥/计费一处管);**待 ADR-004 正式定**。

2. **三个候选拦截点**(生成管线的哪几处过审):
   - world-gen 返回后(整个世界 JSON 落库/下发前一次性审)
   - event-loop 流式中(每回合叙事流式产出时审)
   - 玩家输入侧(玩家自定义输入喂模型前审,防注入/违规诱导)
   三处是否全上、各上到什么强度 → 待定。

3. **流式接缝的两难**(event-loop 侧):
   - **流完再判**:整段叙事生成完再送审 → 审得准、但牺牲逐字流式的即时 vibe(要么等审完再放、要么先放后撤回)。
   - **分句缓冲**:按句/按段缓冲送审、边审边放 → 保住流式手感、但实现复杂、跨句上下文审得糙。
   与 ADR-006(回合叙事先行逐字流)口径的张力如何调和 → 待定。

4. **降级策略**:审核 API 超时/失败时怎么办(放行 vs 拦截 vs 本地兜底词库兜底 vs 保守 no-op)——安全与可用性的权衡,待定。

5. **本地兜底词库**:审核 API 之外的第一道/最后一道本地关键词过滤(§六「接审核 API + 兜底关键词过滤」),词库来源/维护/命中动作 → 待定。

## 六、④ 最小可上线形态 · 倾向与成本闸门

> 形态取舍是决策项,以下为**倾向**;成本闸门为软启动硬门槛,须在放量前落地。

**倾向形态(软启动版)**:

- **不做账号**:软启动阶段不接账号系统(账号 + 云存档是 Phase 2 遗留主线 / 主题 A 持久世界记忆的地基,留后续)。
- **存档走现有本地存档**:复用 Phase 1 已有的本地存档,软启动不引入服务端存档。

**成本闸门(由 ¥200/月 反推)**:

- **全局日限额熔断(硬)**:全站每日 API 调用/成本触顶即熔断,防陌生流量把账单打爆——**软启动放量的硬门槛**。
- **单设备日回合数限(软)**:单设备每日回合数上限,平滑单用户消耗、匀开预算,软限即可(超限提示而非全站熔断)。

> ¥200/月 → 反推日均可承受调用量 → 定全局日限额阈值与单设备日回合上限的具体数字(留 ④ 工作单元按当时真实单回合成本算)。
