## 一、路线选择:A(境内合规)vs B(境外托管)

上线路径有两条互斥的大方向,本次定调选 **B**。

| 维度 | 路线 A · 境内合规 | 路线 B · 境外托管(**本次选定**) |
|------|------------------|-------------------------------|
| 托管地 | 腾讯云 CloudBase 云托管(境内) | 境外容器托管(待 ③ 勘察选型) |
| 备案 | ICP 备案(执照 + 域名实名 + 管局核验) | **不备案** |
| 域名 | `.cn`,营业执照实名 | 个人身份可注册(选型待 ③) |
| 微信生态 | 原生集成(小程序 `callContainer` + 登录/支付/审核) | **暂不进**(纯网页链接) |
| 微信支付 | 可接 | **收不了** |
| 国内访问 | 快(境内线路) | **跨境线路较慢** |
| 微信内分享打开 | 可 | **不能直接在微信内打开**(易被拦/降级为外部浏览器) |
| 行政成本 | 高(执照境内经营地址 + 经营者远程核验) | 低(无行政线) |
| 上线速度 | 慢(卡备案墙钟) | **快**(无备案等待) |

**路线 B 的代价(显式记清,别自欺)**:

1. **国内访问跨境线路较慢**:境外托管对国内用户有跨境延迟/丢包,首屏与流式体感劣于境内。
2. **微信内不能直接分享打开**:分享到微信的链接大概率被拦或强制跳外部浏览器,伤「微信生态分享增长」这条原定获客路径。
3. **收不了微信支付**:变现的微信支付通道不可用(软启动阶段本就免费,影响后置)。
4. **未来仍需补备案**:要冲微信生态 / 收费时,备案 + 执照 + 域名实名这一整套仍得补回来——是**延后不是免除**(迁移成本见 §五)。

**选 B 的理由**:

- **Felix 人在悉尼**:办个体工商户执照需境内经营地址、经营者远程核验成本过高,**时间不够用**。
- **行政成本过高**:路线 A 的墙钟被管局排期与远程核验双重卡住,业余工时下不划算。
- **验证优先**:当前首要目标是**软启动验证产品**(值不值得继续投入),不是规模化 / 收费——路线 B 用最低行政摩擦最快把产品放到真实用户面前。

> 路线 B 是**有意识的择序**:先验证、后合规。境内合规不是放弃,是冻结待解冻(§四腿①)。

## 二、四腿重排(路线 B 下的新优先级)

路线 B 把四腿的优先级整体翻转:原「① 备案优先」降为冻结,「③ 部署」升为当前主线。

| 腿 | 名称 | 路线 B 下的状态 | 依赖 |
|----|------|----------------|------|
| ③ | **境外部署接缝** | **当前主线**(需新 ADR-015 修订/取代 ADR-002 部署形态) | 无硬前置;不再等 ①(不绑 `.cn`、不等备案号) |
| ② | ADR-004 内容安全 | 保留,**优先级降一档** | 无(不依赖其余三腿) |
| ④ | 最小可上线形态 | 决策项;成本闸门**更重要** | 无;成本闸门为软启动**硬门槛** |
| ① | 备案 + 执照 + 域名实名 | **冻结**(记解冻条件) | —— |

**依赖关系图(路线 B)**:

```
③ 境外部署 ── 当前主线,不再等 ① 备案号(用境外托管默认域名/个人注册域名即可跑通)
② 内容安全 ── 独立;审核 API 密钥 + 出网形态仍影响 ③ 密钥管理定型,但优先级降一档
④ 最小形态 ── 独立;「成本闸门」是软启动放量的硬门槛(境外无实名墙,被刷风险更高)
① 备案冻结 ── 解冻条件 = Felix 回国 或 验证通过需冲微信生态/收费
```

**逐条重排理由**:

1. **③ 升为当前主线**:路线 B 下上线的第一堵墙从「行政备案」变成「境外部署接缝」。原 ADR-002 选 CloudBase 的部署形态在境外语境失效(见 §六待决冲突),③ 要先起 ADR-015 定境外托管形态,才能把已验通的 Spring Boot 后端真跑起来。③ 不再被 ① 卡(不需要备案号、不绑 `.cn`)。

2. **② 保留、降一档**:路线 B 下内容安全不再是「境内合规硬要求」(无备案、无管局审查),但**仍要做**,理由变了——
   - **防模型输出伤玩家**:生成内容仍可能产出不适内容,软启动放真实用户进来就有暴露面。
   - **防滥用刷成本**:玩家自定义输入若被用作滥用/越狱入口,直接吃 API 预算(与 ④ 成本闸门同源关切)。
   - **为未来补合规预留可插拔接缝**:现在留 moderation 接缝**零成本**(ADR-002 已划服务边界),以后回国补 TMS 只是填实现;现在不留、以后补是**动手术**。
   优先级降一档 = 排在 ③ 之后,但接缝不能省。

3. **④ 追加两条**:
   - **暂不接微信生态、纯网页链接分享**:软启动靠外部浏览器 + 网页链接触达,不依赖微信内打开。
   - **成本闸门不变且更重要**:境外托管**没有微信实名墙**兜底,匿名流量被刷的风险高于境内——全局日限额熔断从「稳妥」升级为「刚需」。

4. **① 冻结**:备案 + 执照 + 域名实名整套冻结,**解冻条件显式记为**:**Felix 回国** 或 **验证通过、需冲微信生态 / 收费**。冻结不等于删除——是把行政线的墙钟成本推迟到「确认产品值得」之后再付。

## 五、回国补合规的迁移成本评估

路线 B 是「先验证、后合规」,解冻时要补的迁移成本按三块拆,**没有一块是推倒重来**:

1. **行政 = 纯延后不加价**:执照 + 备案 + 域名实名这套行政流程,现在做和回国做**成本相同**——只是时点推后,不因走了路线 B 而变贵。省下的是「悉尼远程办理」的额外摩擦。

2. **工程 = 取决于接缝成色**:[ADR-002](adr/ADR-002-backend-form-factor.md) §最终决策.2 + 关键理由 4 已立**薄适配层纪律**(业务逻辑写成平台无关的 Spring 模块,CloudBase / 微信集成只作薄适配层)。**若纪律成色如约,则迁移 ≈ 搬容器 + 重写适配层、业务核心不动**;**实际成色待 ③ 勘察检验**(③ 首次把后端真跑起来,才验证得了业务逻辑是否真的平台无关)。同理 ADR-004(待起)的 moderation 接缝:现在留(零成本),回国接 TMS 只是填实现接口,不是重构。

3. **数据 = ④ 无账号本地存档顺手消除服务端迁移债**:④ 软启动不做账号、存档走客户端本地存档 → **无服务端用户数据**。解冻迁移时没有「用户库/存档库跨境搬迁」这笔债——数据迁移成本天然为零。

> 一句话:路线 B 的补合规成本 = 行政延后(不加价)+ 工程搬家(接缝守住则非重建,成色待 ③ 验)+ 数据零债(无账号)。

## 七、④ 最小可上线形态 · 倾向与成本闸门

**倾向形态(路线 B 软启动版)**:

- **不做账号**:软启动阶段不接账号系统(账号 + 云存档是 Phase 2 遗留主线 / 主题 A 持久世界记忆地基,留后续)。
- **存档走最小续局([ADR-015](adr/ADR-015-overseas-deployment-form-factor.md) A 项)**(2026-07-16 订正:原「复用现有本地存档」系误记——Phase 1 从未落过本地存档,session 一直是纯内存、刷新即丢):后端每 saveId 一个 JSON 文件落盘(持久卷)+ 前端 localStorage 记 saveId + 续局入口;不做账号、不引数据库。§五.3「数据零债」微调:服务端有匿名存档文件、无用户账号数据,解冻迁移时最多 scp 一个目录,债仍趋零。

**成本闸门(由 ¥200/月 反推;境外无实名墙、更重要)**:

- **全局日限额熔断(硬)**:全站每日 API 调用/成本触顶即熔断——**软启动放量的硬门槛**。境外托管无微信实名墙兜底、匿名流量被刷风险更高,这条从稳妥升为刚需。
- **单设备日回合数限(软)**:单设备每日回合数上限,平滑单用户消耗、匀开预算,软限即可(超限提示而非全站熔断)。

> ¥200/月 → 反推日均可承受调用量 → 定全局日限额阈值与单设备日回合上限的具体数字(留 ④ 工作单元按当时真实单回合成本算)。

**④ 待办挂账**:

- **`prompt_cache_hit_tokens` 观测**(2026-07-16 挂):LLM usage 观测本次只透传三个标准字段;DeepSeek usage 块另带方言字段 `prompt_cache_hit_tokens`/`prompt_cache_miss_tokens`。event-loop 每回合全量回传骨架 → 缓存命中率应较高,而缓存命中价差大、**直接影响 ¥200/月 预算换算**——④ 做成本换算时把这对字段一并解析观测。

   - world-gen 返回后(整个世界 JSON 落库/下发前一次性审)
   - event-loop 流式中(每回合叙事流式产出时审)
   - 玩家输入侧(玩家自定义输入喂模型前审,防注入/违规诱导 / 防滥用刷成本)
   三处是否全上、各上到什么强度 → 待定。

3. **流式接缝的两难**(event-loop 侧):
   - **流完再判**:整段叙事生成完再送审 → 审得准、但牺牲逐字流式的即时 vibe。
   - **分句缓冲**:按句/按段缓冲送审、边审边放 → 保住流式手感、但实现复杂、跨句上下文审得糙。
   与 ADR-006(回合叙事先行逐字流)口径的张力如何调和 → 待定。

4. **降级策略**:审核 API 超时/失败时怎么办(放行 vs 拦截 vs 本地兜底词库兜底 vs 保守 no-op)——安全与可用性权衡,待定。

5. **本地兜底词库**:审核 API 之外的第一道/最后一道本地关键词过滤,词库来源/维护/命中动作 → 待定。**路线 B 下本地兜底词库价值更高**(境外调境内 API 若不稳,本地兜底是第一道防线)。
