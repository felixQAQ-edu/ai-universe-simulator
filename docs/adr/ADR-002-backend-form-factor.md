# ADR-002 · 后端形态选 Spring Boot 运行于 CloudBase 云托管(应用层自控 + 微信原生集成)

- **日期**:2026-06-18
- **状态**:已采纳
- **决策者**:Felix

## 背景

Phase 0 收尾、Phase 1(单模式 H5 闭环)动工前,需正式敲定**后端形态**。这是 ROADMAP §三 标注「对本项目影响最大」的待决议题(ADR-002)。

后端要承担一系列职责,任何形态都绕不开:

- **LLM 调用代理**:隐藏运行模型 API Key(绝不下放前端),做 DeepSeek 上下文缓存友好的转发(承接 [ADR-001](ADR-001-runtime-model-and-provider-abstraction.md) 的 provider 抽象与缓存假设)。
- **内容安全审核网关**:所有生成文本过审后再返前端(CONTEXT §三.7,方案见待写的 ADR-004,第一天即需)。
- **账号与云存档**:Phase 2 起的登录与世界线云存档。
- **计费 / 每日免费额度 / 限流**:Phase 3 变现的基础设施。
- **接口面**:Phase 1 面向 H5,Phase 4 面向微信小程序。

后端形态直接影响**运维负担、微信原生集成、上线速度、生态锁定风险、可迁移的后端学习/求职价值**,需正式决策。

约束条件:

1. **工时是最硬约束**:业余每周 10–20h,范围膨胀是头号敌人,运维负担直接吃掉开发时间。
2. **目标用户在微信生态**:登录 / 支付 / 内容审核的微信原生集成价值高。
3. **平台路线 H5 先行 → Phase 4 小程序(Taro)**:后端要能平滑衔接两端。
4. **内容审核第一天即需**(ADR-004),后端从架构第一天就要留出审核网关边界。
5. 本仓库为**产品仓库**;系统性后端学习另有独立「AI 后端架构师一年计划」仓库。但本决策**显式接纳一定学习/求职价值作为决策因子**——在工时约束下仍想保留可迁移的后端架构实战经验。

决策驱动因素(按权重):① 交付速度与运维负担 → ② 微信原生集成(登录/支付/审核)→ ③ Phase 4 小程序化衔接 → ④ 可迁移的后端学习/求职价值(应用层架构,非部署衬底)→ ⑤ 成本(基建绝对值;注:LLM token 成本远大于后端基建成本)→ ⑥ 生态锁定风险。

## 候选方案

### 方案 A:自建 Spring Boot(通用云)

服务器自购,Spring Boot 应用 + 关系库,部署在通用云(阿里云/腾讯云 ECS 等)。

**优点**:
- 学习价值与可迁移性最高,生态锁定最低。
- 应用层与基础设施全自控,无平台约束。

**缺点**:
- **运维最重**:服务器、证书、ICP 备案、扩缩容、监控全自理——直接撞约束 1(工时)。
- 微信登录/支付/审核**全手搓**,无原生捷径,撞约束 2。
- Phase 4 小程序阶段还要过**合法域名白名单 + 备案**,衔接最慢,撞约束 3。
- 上线速度最慢。**排除**:在业余工时下,基础设施运维的时间黑洞性价比过低,且其学习产出(infra-ops)相对通用、日后易补。

### 方案 B:CloudBase 纯 BaaS(云函数 + 云数据库,Node.js)

微信云开发纯 serverless:云函数 + 云数据库,Node.js 范式。

**优点**:
- **免运维、上线最快**,直接命中约束 1。
- 微信登录/支付/审核**原生打通**,免域名白名单与备案,H5/小程序共用环境(约束 2/3 最优)。

**缺点**:
- 范式换到 Node.js 云函数,**后端架构学习/求职价值低**(撞决策因子 ④)——应用层经验难沉淀为可迁移的服务端架构能力。
- **专有协议 + 专有库锁定最深**:业务逻辑与平台 SDK 深度耦合,迁出代价最高(撞因子 ⑥)。
- **排除**:交付速度虽最优,但牺牲了本决策显式纳入的学习/求职价值,且锁定最深。

### 方案 C:Spring Boot 运行于 CloudBase 云托管(Cloud Run)(本 ADR 采纳)

Spring Boot 应用打成标准容器,跑在 CloudBase **云托管(Cloud Run)**;微信侧经**云托管 + 小程序 `callContainer`** 走私有链路,拿到登录/支付/审核的原生集成与**免小程序域名白名单**。

**优点**:
- **Spring Boot 应用层经验 100% 照搬**:SSE 流式输出、LLM 代理、计费、ChatMemory 等(复用「AI 后端架构师」主线肌肉记忆,命中因子 ④)。
- 经云托管 + `callContainer` 拿到**微信原生集成 + 免小程序域名白名单**(命中因子 ②/③)。
- 应用是**标准容器**,可迁移性显著优于纯 BaaS——换云只需搬容器(缓解因子 ⑥)。
- 免自购服务器/扩缩容/监控,运维远轻于方案 A(缓解因子 ①)。

**缺点**:
- 比方案 B 慢/重:多一层容器构建/部署心智。接受——这是换取学习与求职价值付的**明确溢价**。
- 比方案 A **少练基础设施运维**(扩缩容、监控由平台托管)。接受——infra-ops 通用易补,应用层架构才是简历核心。
- 集成层仍锁微信/腾讯。接受——与产品战略(微信生态)同向,边际风险可控。

## 最终决策

**方案 C — Spring Boot @ CloudBase 云托管**。

### 1. 选 C 而非 A/B 的核心权衡

在工时约束下仍想保留可迁移的后端架构学习与求职价值,**C 是唯一同时满足**「复用 Spring Boot 应用层经验 + 微信原生集成 + 免小程序域名白名单 + Phase 4 顺滑衔接」的路径:

- 简历价值集中在**应用层架构**(AI 管线 / SSE / 计费),C 完整保留;被舍弃的 infra-ops 学习相对通用、日后易补 → C 的**学习产出/工时比优于 A**。
- 主动接受的取舍:C 的交付速度/运维劣于 B,这是换取学习与求职价值所付的明确溢价。

### 2. 缓解锁定的工程约束(沿用 ADR-001 抽象哲学)

ADR-001 已确立「业务管线与 provider 解耦,换 provider 只改配置」的哲学;本 ADR 把同一原则套到平台集成上:

- **业务逻辑写成平台无关的 Spring 模块**,CloudBase / 微信集成只作**薄适配层**。
- **LLM 代理 + 审核网关划为干净服务边界**,可独立迁出(与 ADR-004 交叉引用)。
- **数据层倾向 CloudBase 内置 MySQL** 而非专有文档库,降低迁出成本(见开放问题 1)。

### 关键理由

1. **工时约束下保留学习/求职价值的最优解**:C 是唯一在「免大量运维 + 微信原生集成 + 可迁移应用层经验」上同时不妥协的方案。
2. **复用既有肌肉记忆**:SSE / LLM 代理 / 计费 / ChatMemory 等 Spring Boot 模式直接照搬,不重新学一套云函数范式。
3. **避开两个极端陷阱**:既不掉进方案 A 的运维时间黑洞,也不掉进方案 B 的深度专有锁定。
4. **保留演进路径**:标准容器 + 薄适配层,未来若要迁出 CloudBase,搬容器 + 重写适配层即可,业务核心不动。

## 已知代价

1. **比纯 BaaS 多一层容器心智**:容器构建/部署/镜像管理的认知与时间开销,是为学习价值付的明确溢价。缓解方式:云托管托管了扩缩容/监控,心智集中在 Dockerfile + 部署配置,一次配好后稳定。
2. **集成层锁微信/腾讯**:`callContainer`、微信登录/支付/审核 SDK 与平台耦合。缓解方式:收口在薄适配层,业务核心平台无关;且与产品战略(微信生态)同向,边际风险可接受。
3. **ICP 备案未完全免除**:**H5 阶段对外公网域名仍需 ICP 备案**——备案/白名单的节省主要在 Phase 4 小程序 `callContainer` 阶段兑现。缓解方式:**Phase 1 排期须显式纳入备案**,勿等上线才发现卡备案(见开放问题 3)。
4. **比方案 A 少练基础设施运维**:扩缩容、证书、监控由平台代劳,这部分实战经验拿不到。接受——属通用、易补的能力,非简历差异点。
5. **结论部分建立在「云托管能力」的假设上**:已验证假设——**云托管原生支持 SSE/WebSocket,Spring Boot 流式输出可保留**(承接 ADR-001 的流式诉求);若实测不达预期需回退评估。

## 重新审视的触发条件

- **流式输出实测不达标**:Phase 1 在云托管上跑通 SSE 时,若 TTFT/长连接稳定性明显劣于本地直连(对照 ADR-001 实测 TTFT~1s),重评云托管或 LLM 代理形态。
- **备案/合规卡点**:H5 阶段 ICP 备案或小程序类目/审核出现无法绕过的阻塞 → 重排上线路径甚至后端形态。
- **运维负担超预期**:云托管的容器运维实际吃掉的工时显著超过方案 B 的预期溢价 → 重新权衡是否退回纯 BaaS。
- **成本翻盘**:云托管基建绝对成本异常上升,或与 LLM token 成本量级接近(当前假设 token 成本远大于基建)→ 重核。
- **迁出需求出现**:若产品需脱离微信生态(如出海/独立 App 为主),薄适配层假设受真实迁移检验 → 评估迁移成本是否如预期可控。
- **Phase 4 衔接实测**:小程序 `callContainer` 私有链路与免白名单实际体验,若不如预期顺滑 → 重评跨端方案。

## 实施步骤

> 本 ADR 为形态选型决策,落地随 Phase 1 推进;以下为后续实施骨架。

1. ⏳ 初始化 Spring Boot 工程骨架,**业务逻辑(world-gen/event-loop 管线、计费、存档)与平台适配层分包**,适配层(CloudBase/微信 SDK)单独隔离。
2. ⏳ 打通 CloudBase 云托管:Dockerfile + 部署配置,**实测 SSE 流式输出**(验证已知代价 5 的假设)。
3. ⏳ 划定 **LLM 代理 + 审核网关** 服务边界(与 ADR-004 协同),API Key 仅存后端环境变量。
4. ⏳ 数据层定型:确认 CloudBase 内置 MySQL(开放问题 1),设计存档/账号/计费表。
5. ⏳ **Phase 1 排期纳入 ICP 备案**(已知代价 3),明确 H5 公网域名备案时点。
6. 用 `/roadmap-update` 在 ROADMAP §三/§五 落 ADR-002 决策、ADR 索引移入已完成,并把 ADR-002 从待决议题移除。

## 实际效果(事后补充)

*Phase 1 后端跑通时回填:云托管上 Spring Boot SSE 流式输出的真实 TTFT/稳定性是否与 ADR-001 本地实测一致,容器部署/运维实际吃掉的工时是否落在「明确溢价」的预期内。*

*Phase 4 小程序化时回填:`callContainer` 私有链路与免域名白名单是否如预期兑现,后端环境/账号复用是否真的「换前端即可」。*
