# ADR-008 · 多模式扩展架构:引擎/校验对数值语义无知 + per-archetype 轻量元数据,以「加一个模式」的代价结构为设计目标

- **日期**:2026-06-24
- **状态**:已采纳
- **决策者**:Felix

## 背景

Phase 1(单模式 H5 闭环·规则怪谈)整局闭环已达成并合入 `main`(里程碑「朋友能用手机从头玩到一个完整结局」正式达成)。ROADMAP Phase 2 主线 = **多模式**(末日 / 修仙 / 赛博朋克…复用同一 UG Engine)。架构早留了位:`archetype` 枚举(CONTEXT §三.4)、UG Engine 共用管线(world-gen → char-gen → rule-gen → event-loop → ending-gen)、CONTEXT §三.5 留了「模式特有数值在 schema 扩展定义」的口子——但**未定怎么扩**。

真正要定的不是「能不能加末日」,而是**「加第 N 个模式」的代价结构**:加末日时每个决定,要么让「日后加修仙」变顺,要么变成每模式各写一套的债。末日是第一个落地模式 → 实际在**给所有后续模式立模板**。本 ADR 为 Phase 2 多模式立架构,首个落地模式 = 末日生存(`apocalypse`),签名数值轴 = **饥饿**(它有 hp/san 没有的特性:**随回合自然衰减**,正好压测「模式特有数值能否干净插进共用管线」这个试金石)。

约束条件:

1. **world-gen 是最高风险生成步,每个模式必须独立验**(Felix FINDINGS 反复强调):规则怪谈 5/5 不外推,模式特有数值/规则形态是全新生成面。
2. **复用已验证资产**:Phase 1 的 `Engine.apply`(连推 10 回合三路径自洽)、`validateWorld`/`validateTurn`、状态机、SSE、`toClientState` 消毒、前端整局闭环是护城河,扩模式应最大化复用、最小化核心改动。
3. **golden parity 是第一不变量**:Java 引擎逐字段 == Python(139 测)+ 前端 14 测。加末日**绝不能弄坏规则怪谈**。
4. **数值权威归引擎**(CONTEXT §三.8):AI 提议绝对值,引擎落账;新模式的数值轴不应破坏这条。

### 事实订正(本 ADR 起草时发现,影响决策 1 的措辞)

本 ADR brief 初稿假设「引擎/校验对数值 key 已无知,加模式 `Engine.apply`/`validateWorld`/`TURN_SCHEMA`/状态机/SSE **一行不改**」,理由是「现状就是对 hp/san 做通用处理,不关心语义」。**起草本 ADR、动手前摸代码时发现该前提与实际代码不符**:核心实为 **key-fixed 到 `{hp, san}`**,并非 key-agnostic——`hp`/`san` 是引擎里两个具名 `double` 字段,`validateWorld`/`validateTurn` 各**硬性 require `hp` 与 `san`**,`Engine.snapshot()` 把 `attributes` 整体重建为**恰好 `{hp, san}`**,`EventLoopService.buildDelta` 也硬编码 hp/san。喂 `{hp, hunger}`(无 san)会:① 校验直接挂在「缺失 san」→ world-gen 整局 ERROR;② 引擎丢弃 hunger、注入幽灵 san=100。

「不关心 hp/san 的**语义**」为真,但「对**任意 key** 做字典化处理」为假。故「零核心改动」不成立。Felix 据此发现拍板订正(见决策 1):引擎/校验器/`buildDelta` **一次性从 key-fixed 泛化为 key-agnostic**(遍历 `attributes` map),严格限「字段名→字典遍历」这一种机械变换(结算序列/clamp/跳变/兜底逻辑**一律不动**),由 **golden parity 139 测逐字节守 `{hp, san}` 零回归**。这是一处**有界的核心改动**,不是「一行不改」——本订正记在「已知代价」。

## 候选方案

> 主轴有两组正交决策:**(甲)数值架构**(怎么让模式特有数值插进共用管线)与 **(乙)自然衰减落点**(谁负责让饥饿每回合掉)。下面按这两组列候选。

### 数值架构 · 方案 A:全开放 attributes,无元数据

`attributes` 纯开放字典,引擎遍历 key 通用结算,无任何 per-archetype 声明。

**优点**:
- 引擎零分派、加 key 零成本。

**缺点**:
- **前端不知该渲染哪些数值轴、用什么中文名**(末日玩家看不到「饥饿」);**world-gen/event-loop 提示词不知该生成/维护哪些数值**。纯 A 缺一份「告诉消费方有哪些轴」的轻量声明 → 不可用。

### 数值架构 · 方案 B:每模式声明硬 schema 清单

每个 archetype 在校验器里登记一份硬性 attributes 清单(末日必须 `{hp, hunger}`、规则怪谈必须 `{hp, san}`),`validateWorld`/`validateTurn` 按 archetype 分派对应清单做强校验。

**优点**:
- 校验强:模型漏给 hunger / 给错轴 → schema 直接拦。

**缺点**:
- **加模式要改校验器分派逻辑**——每加一个模式都动核心校验代码,正是要避免的债结构。
- 与「引擎对数值无知」哲学相悖:校验器反而要懂每个模式的数值语义。

### 数值架构 · 方案 C:C 偏 A — 引擎/校验对数值 key 无知 + per-archetype 轻量元数据(本 ADR 采纳)

引擎/校验对数值 key **语义无知**(遍历 `attributes` map 通用结算,范围 0–100 仍硬校验,**key 集合不硬校验**);另立一份 **per-archetype 轻量元数据**(非强制校验)声明每个 archetype 的数值轴清单 + 中文名 + 衰减提示,供两个消费方(前端面板 / 提示词注入)读取。

**优点**:
- **加模式 = 一条元数据 + 一个注入块,不碰引擎/校验/状态机核心**(决策 1 泛化一次到位后)。
- 元数据补上纯 A 缺的「消费方该渲染/生成什么」,又不像 B 那样把模式语义焊进校验器分派。

**缺点**:
- **牺牲硬校验**:模型若漏给 hunger,schema 不拦。校验弱是**有意的代价**;正确性靠提示词约束 + 冒烟门兜。
- **需一次性把现状的 key-fixed 核心泛化为 key-agnostic**(见事实订正)——不是零改动,而是一处有界核心改动,由 golden parity 守零回归。

### 自然衰减 · 方案 甲:引擎落衰减

引擎认识「饥饿是个衰减数值」,每回合自动 `hunger -= decay`。

**优点**:
- 衰减由引擎确定性落账,不依赖 AI 稳定性。

**缺点**:
- **破「引擎对数值无知」**:引擎要懂「哪个 key 会衰减、衰减多少」;且**每加一个衰减型数值都要改引擎分派** → 正是决策 1 要避免的债。**否决**。

### 自然衰减 · 方案 乙:AI 落衰减,引擎保持无知(本 ADR 采纳)

饥饿衰减由 **AI 在 `stateUpdate` 给新绝对值**(符合 CONTEXT §三.8:数值=AI 提议绝对值、引擎落账);元数据的衰减提示(「每回合约 -5~10」)**喂进提示词让 AI 知道该衰减**;**引擎照旧通吃、不读 `decay` 字段、不认识「饥饿会衰减」这个语义**。

**优点**:
- 守引擎无知(决策 1);加任意衰减型数值零引擎改动。

**缺点**:
- **AI 落衰减的稳定性未验**(可能忘记/不稳定)。这是会影响所有「带衰减数值模式」的发现 → 设为末日冒烟的**专门观测项**(决策 5)。

## 最终决策

**数值架构 = 方案 C(C 偏 A);自然衰减 = 方案 乙(AI 落,引擎无知)。** 五条锁定决策:

### 1. 数值架构 = C 偏 A:引擎/校验对数值 key 无知(经一次性泛化)+ per-archetype 元数据

- **引擎/校验/buildDelta 一次性从 key-fixed 泛化为 key-agnostic**(事实订正):把 `Engine` 的具名 `hp`/`san` 字段、`snapshot()` 的 `{hp,san}` 重建、`validateWorld`/`validateTurn` 的 require-hp-and-san、`buildDelta` 的硬编码 hp/san,**机械改为遍历 `attributes` map**。**严格限「字段名→字典遍历」一种变换**:绝对值落账、clamp 0–100、跳变 >40 标记、log/结局/兜底等结算逻辑**一律不动**。判据 = **golden parity 139 逐字节全绿**(`{hp,san}` 零回归),不绿即停。
- **`attributes` 保持开放字典**:`validateWorld`/`TURN_SCHEMA` 对 attributes **不做 per-archetype 硬清单校验**(每个数值轴范围 0–100 仍硬校验,**key 集合不硬校验**)。加模式 = `attributes` 换 key(规则怪谈 `{hp,san}` / 末日 `{hp,hunger}` / 修仙日后 `{灵根,境界}`),泛化后**零引擎改动、零校验核心改动**。
- **per-archetype 轻量元数据(非强制校验)**:一份服务端元数据声明每个 archetype 的(世界观描述 / 数值轴清单 + 中文显示名 + 衰减提示 / 规则形态),**两个消费方**:(a) 前端按它渲染哪些数值面板项及中文名;(b) world-gen / event-loop 提示词按它注入该生成/维护哪些数值。**不进硬校验**——模型漏给某轴时靠提示词 + 软处理,不靠 schema 拦。
- **权衡(明记)**:牺牲硬校验换「加模式零核心改动」。校验弱是有意的代价;正确性靠提示词约束 + 冒烟门兜。

### 2. 自然衰减 = AI 落,引擎保持无知

末日「饥饿」随回合自然衰减:**AI 在 `stateUpdate` 给饥饿新绝对值**,元数据给衰减提示(如「每回合约 -5~10」)**喂进提示词让 AI 知道该衰减**;**引擎照旧通吃、不认识「饥饿会衰减」**(守决策 1 引擎无知)。**否决「引擎落衰减」**(破无知 + 每加衰减型数值改引擎)。AI 落衰减稳定性**设为末日冒烟专门观测项**(决策 5)。

### 3. world-gen 提示词 = 通用骨架(单点)+ per-archetype 注入块

- **通用骨架(固定一处)**:输出 schema、id 约定(F-001:`rules[].id` 整数 / `endings[].id` snake_case 字符串)、消毒硬约束(禁 isTrue/hiddenLogic/正确解法进玩家可见文本)、json_object、`openingNarrative` 字段——**模式无关,单点维护**。
- **per-archetype 注入块(变量)**:从元数据注入 `worldview` / `attributes`(含数值轴 + 衰减提示)/ `ruleForm`。加模式 = 写一个注入块 + 一条元数据。
- **否决「每模式一份完整提示词」**:消毒/id 这种安全/正确性硬规矩每模式重抄 = 抄一次错一次;骨架必须单点。
- **lockstep 纪律**:运行时 Java 副本(`WorldGenPromptBuilder` / `TurnPromptBuilder`)与 `.md` 资产**同步改**(沿用 Phase 1 两次教训:只改 `.md` 运行时失效)。
- **event-loop 提示词**同源注入元数据(回合里 AI 要持续维护/衰减该模式数值),保留 Phase 1 叙事清洁度硬约束。

### 4. init 开放 archetype 入参,前端选择 UI 推迟

- `POST /api/game/init` 的 `archetype` 从写死 `rules_creepy` 改为**真正接受值,校验 ∈ archetype 枚举**(非法 → 400);未激活的 archetype(life_sim 等)先拒/标「未开放」。
- **前端 archetype 选择 UI 推迟**:第一批用 init 入参 + 探针验末日管线能跑通(后端验证);选择 UI 等管线证通后作独立小批(已验证闭环的 UI 改造,低风险)。

### 5. 逐模式独立验 world-gen = 硬验收门

Phase 2 第一批做完末日 world-gen + 提示词后,**必经一次真 key world-gen 冒烟**(仿规则怪谈那次):验首次有效率(对照规则怪谈 ~100%,模式特有不外推)+ 格式完整 + **模式特有数值 hunger 齐** + **AI 衰减稳定性**(决策 2 观测项)+ 泄露。**此关不过不算第一批完成。** 规则怪谈成功不外推。

### 关键理由

1. **加模式的代价结构最优**(呼应约束 1/2):泛化一次到位后,加模式 = 一条元数据 + 一个注入块,不碰引擎/校验/状态机核心;「日后加修仙」顺着同一模板。
2. **引擎无知是正确的抽象边界**:引擎对数值**语义**无知(决策 1)+ 衰减由 AI 落(决策 2),使「带衰减数值的新模式」也零引擎改动;骨架单点维护(决策 3)守住消毒/id 安全规矩。
3. **诚实面对事实订正**:发现核心是 key-fixed 而非 key-agnostic 后,选择「一次性有界泛化 + parity 守零回归」(决策 1),而非把 san 当占位硬塞进末日(语义泄露、defeats the purpose),也非改 brief 绕过——一次决策只动一个变量,泛化只做机械变换。
4. **保留演进路径**:硬校验弱是有意取舍,真出问题(模型漏轴)可后续按需给特定模式加软兜或回到方案 B 的局部分派;不预先过度设计。

## 已知代价

1. **核心一次性泛化(事实订正,非零改动)**:brief 原称「Engine.apply/validateWorld/TURN_SCHEMA/buildDelta 一行不改」基于「引擎已 key-agnostic」的**错误事实假设**;实际核心 key-fixed 到 `{hp,san}`,必须一次性泛化为遍历 `attributes` map 才能通吃 `{hp,hunger}`。缓解方式:**严格限「字段名→字典遍历」一种机械变换**(结算/clamp/跳变/兜底逻辑不动),**golden parity 139 逐字节全绿是唯一判据**(`{hp,san}` 零回归),不绿即停。这是有界的核心改动,一次付清,之后加模式才真零核心改动。
2. **硬校验弱**:模型若漏给某数值轴(如 hunger),`validateWorld`/`TURN_SCHEMA` 不拦(只校验已给轴的 0–100 范围,不校验 key 集合)。缓解方式:提示词约束(元数据注入「该生成/维护哪些轴 + 衰减」)+ 冒烟门(决策 5 验 hunger 齐)兜;真高频漏轴再评估按模式加软兜。
3. **AI 落衰减稳定性待验**:饥饿每回合自然衰减交给 AI(决策 2),稳定性未经真 key 验证(可能忘记/不稳定)。缓解方式:设为末日冒烟**专门观测项**;忘记/不稳定如实记(影响所有衰减型数值模式),据数据决定是否调提示词或换策略。
4. **通用模板首版可能为末日调一两轮**:通用骨架要够「模式无关」,首版抽象未必一次到位。缓解方式:首批落地 + 冒烟读数后允许迭代提示词,不卡架构。
5. **每个新模式仍须独立 world-gen 冒烟**(决策 5,不可省):规则怪谈/末日成功**不外推**到修仙/赛博朋克。这是有意的纪律成本,非缺陷。

## 重新审视的触发条件

- **golden parity 139 在泛化后无法逐字节全绿**:说明泛化触动了结算逻辑(超出「字段名→字典遍历」机械变换的范围)→ 停下复盘,回到「只做机械变换」或重审泛化方案。
- **冒烟显示 AI 落衰减不稳定**(决策 2/3 观测项):末日(及后续衰减型模式)AI 频繁忘记衰减饥饿 → 重评是否需提示词强化、二次调用专门结算衰减、或(最后)局部引擎兜底(但那会触碰引擎无知,需新 ADR)。
- **模型高频漏数值轴**:硬校验弱(代价 2)在真实流量中真成问题 → 评估按模式加软兜或局部回到方案 B 分派。
- **引入混合模式 fusion**(Phase 3):多套设定调和成一套数值轴时,per-archetype 元数据的「单 archetype 一份轴清单」假设可能不够 → 届时重审元数据结构。
- **数值轴需要非 0–100 语义**(如修仙「境界」是离散枚举/分段):当前「范围 0–100 硬校验」假设被打破 → 重审校验口径。

## 实施步骤

> 切分由纯到脏(设计稿 §6);ADR 落档与实现**分开 commit**。

1. **核心泛化(决策 1 事实订正)**:`Engine`(具名字段→attributes map 遍历)、`GameSchemas.validateWorld`/`validateTurn`(require-hp-and-san→遍历已给轴范围校验)、`EventLoopService.buildDelta`(硬编码 hp/san→遍历)。**只做机械变换。**
   ```bash
   mvn -q test   # golden parity 139 必须逐字节全绿(唯一判据,不绿即停)
   ```
2. **per-archetype 元数据结构 + 末日条目 + 规则怪谈条目**(纯数据,下游都依赖):`server/.../archetype/`。
3. **world-gen 提示词重构**(通用骨架抽离 + 末日/规则怪谈注入块,lockstep 运行时副本)+ **event-loop 提示词同源元数据注入**(回合也维护/衰减该模式数值,保叙事清洁度硬约束)。
4. **init `archetype` 入参真正接受 + 枚举校验**(非法 400);下游 world-gen 按 archetype 取元数据 + 注入块。
5. **前端数值面板动态渲染**(按 attributes + 元数据中文名;守 ADR-003 边界;永不渲染 isTrue/hiddenLogic)。
6. **确定性测试矩阵全绿**(不打真实 API):回归护城河(139 + 14 全绿)+ 元数据条目结构 + init 合法/非法/未激活 + world-gen 分派取对注入块 + 前端面板喂末日→「体力/饥饿」、喂规则怪谈→「体力/理智」+ 引擎无知确认(`Engine.apply` 对 `{hp,hunger}` 与 `{hp,san}` 行为一致)。
7. **末日 world-gen 真 key 冒烟(决策 5 验收门,Felix 在场)**:首次有效率 / 格式 / hunger 齐 / **AI 衰减稳定性** / 泄露 / 存末日真实 raw + SSE 转录。**此关不过不算第一批完成。**
8. 冒烟通过后 `/roadmap-update`(Phase 2 行 🟨)+ CONTEXT 回写(§三.5 展开 + 元数据结构,`schemaVersion` 仍 "0.2",CONTEXT 文档版本升)。
