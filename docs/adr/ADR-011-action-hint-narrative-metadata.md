# ADR-011 · 选项风险提示为叙事元数据(引擎不裁决)——#1 选择反馈定性版

- **日期**:2026-07-01
- **状态**:已采纳
- **决策者**:Felix

## 背景

Phase 2 打磨期 scoping(2026-06-30)把玩家视角十条反馈坍缩成四块主题,本轮范围含 **#1 选择反馈**(定性版):让玩家在按下决策圈某个选项前,对该选项的**风险/代价/张力**有个定性预期,而不是盲选。目标是「让选择有重量」(主题 B),与 #3 数值行为化同批把四个世界做厚。

**现状勘察结论(接入面已通,唯一缺提示词引导)**:

- schema `availableActions[].hint` 字段**端到端已存在**——前端 `AvailableAction.hint`(`web/src/types/schema.ts`)、wire、`GameSchemas` 对 `hint` 走 `optionalString` 容忍(给了校验字符串、不给放过)、前端 `DecisionCircle` 已条件渲染(`a.hint && …`)。
- 引擎对 `availableActions` 是**纯 `deepCopy` 透传**(不解读、不裁决),`hint` 随选项原样流到客户端。
- 因此 **#1 = 把已有的做厚**:字段/wire/校验/渲染都在,唯一缺的是**提示词从没引导 AI 稳定为每个选项写一句 hint**——world-gen 的 `hint 可空`、event-loop 完全没提 hint。

**为什么现在需要一条 ADR**:#1 触及一个容易失控的产品/架构边界——「给玩家选项反馈」很自然会滑向「显示成功率数字 / 引擎按概率掷骰决定结果」。本轮**刻意只做定性版、不做掷骰**,需要把这条边界**立字为据**(引擎只读透传、永不据 hint gate/掷骰),避免后续实现或加世界时无意破界。这是一条**范围/边界 ADR**,不动 schema / 引擎 / `schemaVersion`。

约束条件:

1. **不动 schema / 引擎 / `schemaVersion`**:`hint` 字段与 wire 已存在,本轮纯提示词 + 前端样式 + 文档;`schemaVersion` 保 `"0.4"`,`EngineGoldenTest` 字节级零回归(引擎一行不动)。
2. **守 ADR-008 引擎无知**:引擎不解读 hint 语义、不据它做任何判定;hint 只是**叙事元数据**,流向玩家、引擎透传。
3. **守三视图消毒(CONTEXT §三.9)**:hint 是玩家可见字段,**绝不携带 `isTrue` / `hiddenLogic`**——提示词硬约束仍管着它(hint 与 `content`/`openingNarrative` 同属玩家可见,泄露硬约束一体适用)。
4. **不牺牲 A-1 叙事长度**:加 hint 指令是给**选项**加一句短提示,不得顶爆正文 2–4 句 / 约 280 字硬上限。

## 候选方案

> 正交的两个问题:**(甲)hint 承载什么**(定性 vs 定量概率)与 **(乙)引擎对结果是否据此裁决**(纯叙事 vs 掷骰结算)。本 ADR 在这两个问题上都选「轻的那一侧」。

### 方案 A:不做(维持现状,hint 偶发)

AI 偶尔给 hint、多数不给,前端有则显、无则不显。

**优点**:零改动。

**缺点**:选择依旧「盲」——多数选项没有任何风险预期,#1 反馈未被满足。**排除**(等于不做 #1)。

### 方案 B:定量成功率 + 引擎掷骰结算

hint 里带精确成功率(如「成功率 60%」),AI 定概率、引擎按概率掷骰决定选项结果、再落数值结算。

**优点**:选择「有重量」最彻底——真随机、真后果,博弈感强。

**缺点**:
- **引擎要认识「概率」并据此裁决结果**——直接违 ADR-008 引擎无知(引擎从「只读透传」变成「按 hint 掷骰、决定叙事走向」),是承重架构改动。
- 依赖**状态层**(掷骰结果要进 state、影响后续结算),Phase 2 打磨期尚无此接缝。
- 精确百分比一旦显示,玩家会**据数字反推最优解**、把体验从「读氛围做抉择」拉成「刷概率」,反而削弱沉浸。
- **排除本轮**:是独立的大工作单元(schema + 引擎 + 状态层 + ADR),记为 **#1.5「选项真概率掷骰」**冻结入 backlog,不在本轮。

### 方案 C:定性风险提示为叙事元数据,引擎只读透传(本 ADR 采纳)

hint = 一句**定性**的风险/代价/张力提示(如「天劫已锁定你」「恐损道基」「可能引来镇民」),**不含精确成功率数字**;AI 在 world-gen + event-loop 为每个选项写;引擎**只读透传、永不据 hint gate/掷骰**;前端把 hint 作选项文字下方的独立小字展示。

**优点**:
- **零架构改动守边界**:引擎依旧纯 `deepCopy` 透传(ADR-008 不破),schema/`schemaVersion`/引擎全不动。
- **定性提示给「重量」但不给「最优解算法」**:玩家有预期、要权衡,但读到的是氛围化的风险话术而非可反推的数字。
- **复用已通的接缝**:字段/wire/校验/渲染都在,唯一缺的提示词引导补上即可,是「做厚」不是「加功能」。
- **留演进路径**:若日后真要博弈深度,#1.5(定量掷骰)是明确的后续档,本轮定性版是它的自然前身。

**缺点**:
- hint 是**软产出**——AI 可能偶发漏写或写得空泛(全靠提示词引导,无硬校验兜)。缓解:world-gen + event-loop 四处 lockstep 明写「每个选项写一句定性风险/代价」+ 冒烟门验四世界稳定出现。
- 定性提示**不是真概率**:玩家若期待「数字化胜率」会落空。这是刻意的——本轮就是不做掷骰(见 (乙))。

## 最终决策

**方案 C — 定性风险提示为叙事元数据,引擎只读透传。** 核心边界(**逐字锁定**):

> **hint = 定性风险/代价元数据、不含精确成功率数字、引擎只读透传永不 gate/掷骰;真概率掷骰 = #1.5 独立未来项,不在本轮。**

### 1. hint 语义与边界

- **是什么**:每个 `availableActions[].hint` 是**一句定性的风险/代价/张力提示**——点出选这个选项**可能付出什么代价 / 面临什么风险 / 有什么取舍张力**(如「天劫已锁定你」「恐损道基」「可能引来镇民注意」「越靠近,那低语越清晰」)。
- **不是什么**:**不含精确成功率数字 / 百分比 / 掷骰概率**(不写「成功率 60%」「30% 触发」)。定性话术,氛围化,贴合本模式口吻。
- **hint 是叙事提示,不代表引擎会据此判定**:引擎**只读透传**(纯 `deepCopy`),**永不据 hint gate 选项、永不掷骰、永不据它改数值结算**。选项合法性仍只按 id 校验(现状),结果仍由 AI 叙事 + `stateUpdate` 落账(现状)。
- **消毒**:hint 是玩家可见字段,**绝不携带 `isTrue`/`hiddenLogic` 或规则真伪/正确解法**(与 `content`/`openingNarrative` 同属泄露硬约束管辖面,三视图消毒不破)。

### 2. 提示词引导(本轮核心改动,lockstep 四处)

让 AI 在 **init(world-gen)+ turn(event-loop)两处**稳定为每个选项写一句定性 hint:

- **world-gen**:`prompts/world-gen.md` + `WorldGenPromptBuilder.java`——把 `availableActions[].hint 可空` 升级为「**每个选项写一句定性的风险/代价/张力提示**(示例:『天劫已锁定你』『损道基』『可能引来镇民』),**不写精确成功率数字**」。
- **event-loop**:`prompts/event-loop.md` + `TurnPromptBuilder.java`——现在完全没提 hint,**补上同一条 hint 指令**(每回合的 `availableActions` 每个选项写一句定性 hint,同上边界)。
- **四处 lockstep 逐条对齐**(沿用「只改 `.md` 运行时失效」的清洁度修法),两个 `.md` 各升版号;明文写「hint 是叙事提示,不代表引擎会据此判定」呼应本 ADR 不掷骰边界。

### 3. 前端展示

`DecisionCircle.tsx`:hint 作**选项文字下方的独立小字**(muted、次级层级),缺失时不显不占位(条件渲染已在,纯样式/布局)。纯展示、不改取数。

### 关键理由

1. **守 ADR-008 引擎无知**:引擎对 hint 零解读、零裁决——从「已 `deepCopy` 透传」到「继续 `deepCopy` 透传」,一行不动。「让 AI 写 hint、引擎不看」是守无知的关键(同 ADR-010「让 AI 标 outcome」)。
2. **做厚而非加功能**:字段/wire/校验/渲染四层皆在,唯一缺的提示词引导补上即可——最小变换、`schemaVersion` 不动、golden 字节级零回归。
3. **正确避开「显示概率 → 玩家刷最优解 / 引擎掷骰破无知」双陷阱**:定性话术给重量、不给算法;掷骰整包推迟到 #1.5,不在本轮无意开这个口子。
4. **保留演进路径**:#1.5(AI 定概率 + 引擎掷 + 状态结算)是明确的后续档,本轮定性版是其前身;届时另起 schema + 引擎 + 状态层 + 新 ADR。

## 已知代价

1. **hint 是软产出、无硬校验兜**:AI 可能偶发漏写某选项 hint 或写得空泛——全靠提示词引导。缓解方式:四处 lockstep 明写「每个选项写一句」+ 冒烟门验四世界每局每个选项稳定出现且贴合;`GameSchemas` 对 hint 仍 `optionalString`(不给容忍,不因漏写而校验失败拖垮整局)。
2. **定性 ≠ 真概率,选择「重量」有上限**:玩家读到的是氛围化风险话术、非可博弈的数字胜率,深度不如掷骰。缓解方式:这是**刻意的本轮边界**,#1.5 冻结入 backlog 承接真博弈需求;定性版先验「选项反馈有没有帮到抉择感」。
3. **多一句 hint 占用 token 与叙事注意力**:每个选项加一句短提示,轻微增本、也占玩家阅读。缓解方式:hint 是**短句**(非整段),world-gen/event-loop 骨架已有长度纪律(A-1 管正文;hint 明写「一句」),冒烟验不顶爆正文。
4. **hint 泄露风险面(玩家可见字段)**:hint 写不好可能剧透隐藏机制。缓解方式:hint 纳入既有泄露硬约束(与 `content`/`openingNarrative` 同管辖),提示词明写不带 `isTrue`/`hiddenLogic`;消毒测试覆盖 hint 路径永不泄露。

## 重新审视的触发条件

- **冒烟显示定性 hint 不够、玩家仍想要数字胜率 / 真后果**:选择「重量」定性版不足以支撑抉择感 → 启动 **#1.5**(定量掷骰,独立 ADR + schema + 引擎 + 状态层)。
- **AI 频繁漏写或写空泛 hint**:提示词引导压不住(类比 ADR-010 A 软引导压不住极性)→ 重评是否需要更强引导或结构化兜底(但那可能触碰校验/引擎,需补 ADR)。
- **hint 泄露隐藏逻辑**:玩家可见 hint 里出现规则真伪/正确解法 → 收紧提示词消毒约束、补消毒测试。
- **状态层落地(账号/云存档/持久世界记忆之后)**:#1.5 掷骰依赖的状态接缝就位 → 重审定性版是否升级为定量版。
- **混合模式 fusion(Phase 3)**:多套设定调和时,单选项一句 hint 的假设可能不够(选项张力来自多世界交叉)→ 届时重审。
