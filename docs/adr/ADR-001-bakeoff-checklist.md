# ADR-001 · Provider Bake-off 验证清单(草稿)

> 用途:Phase 0 核心验证。把本文件交给 Claude Code,让它实现一个对比测试脚本,
> 用同一套提示词 + 同一套场景,横评候选运行模型,产出可复核数据,作为 ADR-001 的决策依据。
> 约定真理之源以 `docs/CONTEXT.md` 为准(schema / 管线 / 命名),本清单只新增"怎么测",不改约定。

---

## 0. 一句话目标

证明(或推翻)"某个运行模型能稳定生成一个好玩、自洽的规则怪谈世界,并连推 10 回合不崩",
并据此敲定:**主力 provider + 兜底 provider + 是否分层 + 回填验收标准实测值**。

---

## 1. 范围与非范围

**本次范围**
- 只测 **run-time 内容生成**(给玩家用的),不评估 build-time 写代码。
- 只测 **规则怪谈 `rules_creepy`**(MVP archetype),**单体模式**(`mode: single`)。
- 管线步骤覆盖:`world-gen` → `char-gen` → `rule-gen` → `event-loop` → `ending-gen`。

**明确不做(防范围膨胀)**
- 不测 `fusion` / 混合模式(那是 Phase 3)。
- 不测其它 archetype(人生/修仙/赛博朋克/末日)。
- 不做前端、登录、云存档。
- 内容安全审核网关本次**只观察、不实装**(为 ADR-004 留素材)。

## 3. 待测指标(按重要性排序)

> **校准说明(2026-06-17,据 DeepSeek V4-Flash 实测重定)**:原"通过线"为拍脑袋估算,
> 现按首跑实测 + 修订后口径重新校准为下表"校准后通过线"。关键口径变化:
> ① JSON 有效率以**修复后**为硬门(系统本就带一次修复重试),首次率改作软指标/成本信号;
> ② 自洽性采用 F-003 修订口径(允许有据恢复,只查跳变/自洽,不再禁任何回升);
> 实测明细见 [bakeoff/out/report.md],发现日志见 [bakeoff/FINDINGS.md]。

| # | 指标 | 怎么测 | 校准后通过线 | DeepSeek V4-Flash 实测 |
|---|---|---|---|---|
| 1 | **JSON 有效率** | 每次生成跑 schema 校验;失败则一次"修复重试" | **修复后 ≥ 99.5%(硬门)**;首次 ≥ 90%(软,越高越省) | 修复后 100% ✅ / 首次 86.8%(F-004 收敛后预期 95%+) |
| 2 | **连推 10 回合自洽不崩** | 跑满 10 回合,核规则一致/属性结算自洽/剧情连贯(允许有据恢复) | 10 回合无逻辑矛盾、无 >40 异常跳变 | 三路径 0 矛盾 ✅ |
| 3 | **hiddenLogic 不泄露** | 扫描所有"玩家可见"文本,确认不含 `isTrue`/`hiddenLogic` 内容 | 0 泄露 | 0 泄露 ✅ |
| 4 | **中文叙事质量** | 人工按下方 rubric 打分(盲评,隐去 provider) | 主观均分 ≥ 4.0(见 §6) | 待人工盲评 ⏳ |
| 5 | **首字延迟 TTFT** | 流式调用,记录首 token 到达时间 | ≤ 2s(冲刺 ≤ 1s) | ~1.05s ✅ |
| 6 | **完整回合延迟** | 一回合叙事 + JSON 全部返回耗时 | ≤ 10s | ~5.4s(σ 3.3) ✅ |
| 7 | **单回合成本** | 输入(命中/未命中分列)+ 输出 token × 单价 | **≤ ¥0.01/回合**(实测留 ~5× 余量,较原估算大幅收紧) | ~¥0.002 ✅ |
| 8 | **缓存效果(DeepSeek 专项)** | 同一存档连续回合,统计缓存命中率与命中/未命中成本差 | 观察项,验证省钱假设 | 命中率 ~34–40% 👀 |

> #4 待人工盲评,#8 为观察项。**DeepSeek V4-Flash 作为 event-loop 主力候选,工程指标已基本通过。**

## 4. Prompt 场景(test cases)

> 提示词统一放 `prompts/`,按管线步骤组织(对齐 CONTEXT 约定 6):
> `world-gen` / `char-gen` / `rule-gen` / `event-loop` / `ending-gen`。
> 输出必须是符合 `docs/CONTEXT.md` §二 schema(v0.1)的 JSON。

### 场景组 A · 世界生成质量(world-gen + char-gen + rule-gen)
一次性生成完整 `world` + `character` 初值 + `rules`(真假混合)。
- **A1 稳定性**:固定种子输入,跑 5 次,看 schema 合法率、规则结构稳定性、方差。
- **A2 多样性**:换 3 个不同种子,看世界是否雷同、规则是否套路化。
- **A3 规则质量**:人工核——规则是否真假混合、`hiddenLogic` 是否自洽、危险等级与基调是否匹配。

种子输入示例(可直接用作 A1):
```
模式:规则怪谈(单体)
场景种子:雨夜便利店,玩家是临时夜班店员,凌晨 0:00–6:00
危险等级:high
要求:生成 6–8 条规则(真假混合),hp/san 初值,2–3 个结局条件
```

### 场景组 B · 连推 10 回合(event-loop)—— **核心**
用 A1 产出的某个固定世界作为起点,脚本模拟玩家选择,连推 10 回合。
每回合回传 `state`(必要时含 `logSummary`),验证模型在无记忆前提下保持一致。
设计 3 条不同玩家路径:
- **B1 求生路径**:每回合都选"遵守已知规则"的行动。
- **B2 作死路径**:故意踩规则,验证规则触发与数值结算是否一致、是否合理走向坏结局。
- **B3 探索路径**:主动试探,验证 `rules[].discovered` 翻转逻辑、新线索是否前后不矛盾。

每条路径要核:① 属性(hp/san)结算前后无矛盾;② 已触发规则后续保持一致;
③ `availableActions` 每回合 2–4 个且合法;④ 玩家可见文本不泄露隐藏字段。

### 场景组 C · 鲁棒性 / 边界
- **C1 非法行动**:玩家提交不在 `availableActions` 里的输入,看引擎/模型如何兜底。
- **C2 摘要压缩**:把 `log` 撑到触发 `logSummary` 压缩后,验证模型仍记得关键事实(规则、关键道具、伤亡)。
- **C3 内容安全观察**:故意往暴力/危险方向推,记录生成内容(只观察,为 ADR-004 留样本,不实装审核)。

### 场景组 D · 结局收敛(ending-gen)
让 B 中某条线走到结局条件,验证:结局判定是否合理、是否正确置 `state.status = ended`、
是否命中预设 `endings[].condition` 并置 `reached = true`。

## 5. 工程实现要求(给 Claude Code)

1. **统一客户端**:一个 OpenAI 兼容 client,读 **provider 配置表**
   (`baseUrl` / `model` / `apiKey` / `price` / 思考模式开关 / `maxContext`)。换 provider 只改配置。
2. **思考模式差异**:Qwen `enable_thinking`、GLM `thinking`、DeepSeek V4-Flash 思考/非思考——
   非标参数走 `extra_body`,在适配器里收口,不污染主流程。
3. **状态机**:严格按 CONTEXT 约定——`state` 是真理之源,每回合回传;旧 `log` 超阈值压缩进 `logSummary`。
4. **每次调用记录**:provider、model、输入 token(命中/未命中分列)、输出 token、TTFT、
   总延迟、原始响应、解析结果、schema 校验结果、估算成本、是否触发修复重试。
5. **可复现**:固定 temperature 与种子;同场景多跑取均值 + 方差。
6. **产出两份文件**:
   - 明细数据(JSON 或 CSV,每次调用一行)
   - 汇总报告(provider × 指标 的对比表 + 通过/未过标注)
7. **泄露检测**:渲染给"玩家可见"的字段时,断言不含 `isTrue` / `hiddenLogic`。

## 6. 中文质量人工评分 rubric(盲评)

每个世界 / 回合按 1–5 打分,隐去 provider:
- 沉浸感与氛围(规则怪谈是否够"瘆人")
- 规则设计巧思(真假混合是否有解谜乐趣)
- 叙事连贯与逻辑自洽
- 玩家抉择的有意义程度(选项是否都像样、有取舍)
- 中文文笔(是否生硬、是否有翻译腔)

**阈值建议**:综合均分 ≥ 4.0 视为"叙事质量达标";低于 3.5 的 provider 即便便宜也不做主力。

跑完后产出一张 **provider × 指标** 对比表,据此回填 ADR-001:
- **Decision**:主力(建议 DeepSeek,Flash 跑 event-loop / Pro 跑 world-gen)+ 兜底(建议 Qwen)+ 是否分层。

## 8. 建议执行顺序(控制业余时间投入)

1. 先搭客户端 + 配置表 + 调用日志(1 个 provider 跑通)。
2. 写 world-gen / event-loop 提示词,跑场景组 A + B(只用 DeepSeek V4-Flash)。
3. 一个 provider 跑顺后,再横向加 Qwen / GLM 跑同样场景。
4. 最后补场景组 C / D 与可选 provider。

> 任何一步发现"连 10 回合都崩",优先改提示词与状态回传,而不是急着换模型——
> 多数自洽问题是 prompt + state 设计问题,不是模型能力问题。
