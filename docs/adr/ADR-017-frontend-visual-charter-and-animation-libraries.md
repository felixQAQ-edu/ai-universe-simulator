# ADR-017 · 前端视觉宪法与动画库许可名单——放开动画库限制,立「界面是世界的一部分」为视觉基准(修订 ADR-003)

- **日期**:2026-07-22
- **状态**:已采纳
- **决策者**:Felix

## 背景

上线阶段四腿收口到 **③ 部署 + ④ 成本闸门完成、① 备案冻结、② ADR-004 待启**(见 [phase3-launch-plan.md](../phase3-launch-plan.md)),万界已上公网(https://wanjie-ai.fly.dev)。新方向拍板:**前端视觉升级升为当前主线**,Felix 亲自玩 + 打磨体验优先——即从「功能能跑通」转向「体验值得留下」。

**与 ADR-003 的关系(修订、非推翻)**:[ADR-003](ADR-003-frontend-stack-and-taro-boundary.md) 的核心是**接口纪律占住 Taro 迁移边界**(平台无关层 + `api/` 薄适配层 + provider-agnostic `TurnStream`),这部分**继续全额有效、本 ADR 一字不动**。本 ADR 只修订其**样式与库选型条款的适用范围**——理由随路线 B 降级:

- ADR-003 §5 选「原生 CSS / CSS Modules」的**唯一论据是「Taro 迁移最安全」**;
- 路线 B([ADR-015](ADR-015-overseas-deployment-form-factor.md) 已落地)把**微信生态 / 小程序整线冻结**,该论据的承重对象随之消失;
- 这与 **ADR-002 被路线 B 激活预登记迁出触发条件**同构——**属预案内演进,非翻案**(本项目第二例)。

**口径出处如实记录(勘察发现,与常见转述不符)**:「纯 CSS、不引动画库」这条禁令**从未写进 ADR-003 正文**。ADR-003 §5 的字面禁令只有一条 =「**不引入运行时 CSS-in-JS**」,它甚至明说「`frontend-design` skill 此阶段可用于视觉方向」。动画库禁令的实际立字散落三处:

| 出处 | 原文性质 | 状态 |
|---|---|---|
| [ADR-003](ADR-003-frontend-stack-and-taro-boundary.md) §5 | 只禁运行时 CSS-in-JS;「Taro 迁移最安全」是**精神来源** | 本 ADR 修订其适用范围 |
| [phase2-A-archetype-select-ui.md](../phase2-A-archetype-select-ui.md) §backlog(2026-06-25) | **动画库禁令首次立字**,自称「违 ADR-003 跨端边界」= 据 §5 精神做的**实践外推** | 本 ADR 取代 |
| [ADR-013](ADR-013-hybrid-fusion-protocol.md) 前端约束一行 | 「不引动画库(纯 CSS,Taro 可复用)」= 复述 | 本 ADR 取代 |

**三处旧文档保持历史原样不回改**(循本项目「旧 ADR 不改写、由新 ADR 修订」惯例,如 ADR-015 修订 ADR-002);**自本 ADR 起,动画库口径以本 ADR 为准**。

**佐证:预登记条件已命中**。A 计划设计稿那条禁令**自带重评触发条件**:「若 Phase 2/3 出现动效密集需求(分享长图炫酷动画、混合模式视觉特效),重评 GSAP——但须先确认其在 Taro/小程序侧可行性」。现在**触发条件命中**(视觉升级升主线),且**「须先确认 Taro 可行性」这个前提因小程序线冻结而失效** → 重评的两个闸门同时打开。

**工程佐证:两条纪律正交**。`web/eslint.config.js` 的硬线**只禁平台 IO 全局对象**(`fetch`/`EventSource`/`WebSocket`/`XMLHttpRequest`/`wx.*`),**不约束任何 npm 依赖**——引动画库在工程上**根本不触碰接口纪律硬线**。「修订库限制 / 接口纪律不动」的切分是干净的,不是措辞上的和稀泥。

**基线现状**:`web/package.json` dependencies 仅 `react` / `react-dom` / `zustand`,**零动画依赖**;现有视觉(顶部氛围带 + 白底正文 + 每世界 CSS 卡片主题 + `prefers-reduced-motion` 关运动)全为手写 CSS。

约束条件:

1. **内容永远第一**:本产品是**阅读型**产品,叙事文本是主体;任何视觉手段不得损害阅读。
2. **接口纪律不受影响**:ADR-003 §2/§3 边界继续有效,视觉层改动限于展示层,逻辑/状态层零改。
3. **引库即欠债**:小程序线只是冻结不是取消,引入的每个库都是未来重做视觉层的债,须显式记账。
4. **范围可控**:视觉升级是打磨轮不是重写,须有明确的第一版边界,防范围膨胀(业余每周 10–20h 的头号敌人)。
5. **不做 AI 工具审美**:视觉必须服务「进入世界」,而不是展示动画能力。

## 候选方案

### 方案 A:维持现状,继续纯 CSS 手写

**优点**:
- 零新依赖、零包体增长、零移植债,小程序线解冻时视觉层几乎可直接复用。

**缺点**:
- 目标是「体验值得留下」,而手写 CSS 在**时间线编排**(多元素有序演出:起局揭幕、结局收束)与**可中断的连续插值**(数值 80→65 的滚动、状态驱动的进出场)上成本极高、可维护性差——这两类恰是本轮三个记忆点的核心。
- 继续以「Taro 迁移最安全」为由约束,而该理由的承重对象(小程序线)已冻结 → **用一条失效理由锁住主线目标**。**排除。**

### 方案 B:全面放开,按需引任意库(含 Three.js / WebGL)

**优点**:
- 天花板最高,视觉表达不受限。

**缺点**:
- 无名单 = 无预算,每次「想要个效果」都可能新引一个库,包体与移植债不受控(撞约束 3/4)。
- Three.js 对**阅读型文字产品**投入产出比不成立:包体百 KB 量级起、低端机与微信 WebView 性能风险高、开发时间成本大,换来的是与「读文字」几乎不相关的 3D 表达。
- 无视觉基准的放开极易滑向「AI PPT 审美」(紫色玻璃霓虹渐变堆砌),撞约束 5。**排除。**

### 方案 C:白名单三库 + 缓引清单 + 视觉宪法(本 ADR 采纳)

**核心原则:放开的是手段,收紧的是标准。** 库按白名单放开(第一批仅三个,一次性拍死),同时把「什么是好」写成可援引的视觉宪法与优先级,让每个效果都能被问一句「它服务哪个原则」。

**优点**:
- 拿到时间线编排与声明式动效两把关键工具,直击本轮三个记忆点(命中约束 4 的「有边界的升级」)。
- 名单制让移植债**可数**(三个库,不是「若干」),回国解冻时算账对象明确(命中约束 3)。
- 视觉宪法 + 优先级 + 反面清单把「好看」从口味问题变成可援引标准(命中约束 1/5)。
- 接口纪律与 eslint 硬线完全不动(命中约束 2)。

**缺点**:
- 白名单需人守——想引第四个库时须回窗口对齐,有流程摩擦。**接受**:摩擦正是防范围膨胀的机制本身。
- 视觉宪法不像 eslint 可自动检查。缓解:原则 ① 是**可判定**的(动画是否影响阅读,冒烟一眼可验),其余三条靠 Felix 冒烟拍板。

## 最终决策

**方案 C — 放开动画库限制到一份三项白名单,同时立视觉宪法为此后所有视觉工作的援引标准。**

### 1. Vision Brief(视觉工作的总纲)

> **AI Universe Simulator 不是一个拥有多个主题的 AI 工具,而是一座通往不同宇宙的入口。界面本身应该像世界的一部分,而不是世界之外的控制台。所有视觉效果都必须服务于「进入世界」和「沉浸其中」两个目标,而不是为了展示动画能力。**

### 2. 视觉宪法(四原则,此后所有视觉决策的援引依据)

1. **内容永远第一,任何动画不能影响阅读。**
2. **动画不是为了炫,而是为了表达。**
3. **每个世界都有自己的视觉语言**——玩家不看标题就知道进了哪个世界。
4. **追求记忆点,不追求炫**——「世界会变化,而不仅仅是页面会动画」。

**优先级配比**:**氛围沉浸(50%)> 交互反馈(35%)> 关键时刻(15%)**。资源与注意力按此分配——氛围是玩家每一秒都在看的东西,关键时刻一局只出现两三次。

**反面清单(一句)**:不做「**AI PPT 审美**」——紫色玻璃拟态、霓虹渐变堆砌、无意义的通用炫技动效。

### 3. 库许可名单(第一批,仅此三个)

| 库 | 承担什么 | 为什么是它 |
|---|---|---|
| **GSAP** | 时间线编排:关键时刻转场(起局揭幕 / 结局收束)、境界跳变整页照亮的多元素有序演出 | 时间线是手写 CSS 最贵的能力;GSAP 是该能力的行业标准件 |
| **Motion(Framer Motion)** | React 声明式动效:选项按压/发光、组件进出场、数值滚动渐变、规则点亮仪式感 | 与 React 状态模型同构,交互反馈层(35%)写起来最省 |
| **Lenis** | 平滑滚动 | 阅读型产品的滚动质感直接作用于原则 ①,单点小件、代价极低 |

**缓引清单**(记录理由,**不引**):

| 候选 | 缓引理由 |
|---|---|
| **Particles**(tsparticles 一类) | 灰尘 / 孢子 / 风沙的第一版**先用 CSS + GSAP 做**,验证不够再引 |
| **Lottie** | 依赖**尚不存在的动画素材**(需 AE/设计产出),现在引 = 引了没得播 |
| **轻量 Shader** | 非第一版;氛围层先做到 CSS + GSAP 的天花板再谈 |

**升级条款**:缓引清单转正**须回 Project 窗口对齐**,**不得在实现批里自行引入**。白名单之外的任何库同此。

**Three.js:明确不做。** 阅读型产品,投入产出比不成立(包体 / 低端机与微信 WebView 性能 / 开发时间,换来的表达与「读文字」几乎不相关)。这不是缓引,是排除。

### 4. 移植债(显式记账)

**引这三个库 = 未来回国做小程序时视觉层重做的债。** Taro 侧非 DOM 渲染,GSAP/Motion/Lenis 均不可直接复用 → 届时小程序端视觉层需按「克制版」重做(退回 CSS 动画 + 小程序原生能力)。

- **债的规模可数**:三个库,不是「若干」;且债**只在展示层**——逻辑/状态/`api/` 层不受影响,ADR-003 接口纪律仍保证「迁移时只换适配层」对**数据链路**成立。
- **挂钩关系**:此债与 ①(备案 + 执照)② 冻结项同属「回国解冻时一并算账」的一篮子;解冻评估须把「小程序视觉层重做」列为显式工作项,不得默认它免费。
- **减债习惯(非硬性)**:能用 CSS 达到的效果优先用 CSS;库只用在 CSS 明显不划算处(时间线、连续插值)。

### 5. 每世界视觉语言表

原则 ③ 的落地依据。**赛博朋克不进表**(未实现世界,配色留到该世界真正上架时定,避免空头设计)。

| 世界 | 主色调 | 光 / 动 | 材质 / 粒子 |
|---|---|---|---|
| 修仙 `cultivation` | 白金青 | 云雾 | 灵气 |
| 规则怪谈 `rules_creepy` | 蓝灰 | 灯闪 | 灰尘 |
| 克苏鲁 `cthulhu` | 墨绿 | 扭曲 | 漂浮孢子 |
| 末日生存 `apocalypse` | 黄褐 | 风沙 | 灰烬 |
| **融合世界**(识海遗蜕 / 守则即补给) | — | — | 具体视觉语言**待 Cowork 探索时提案、回窗口定**;**原则 = host 世界基调打底 + 渗漏元素点缀**(与 [ADR-012](ADR-012-hybrid-axis-merge-strategy.md) 「host 优先 + 语义换皮」同构) |

### 6. 第一版范围(Cowork 探索的边界)

1. **氛围层**:四大世界 + 两融合世界的色调 / 微动态 / 呼吸感——**克制**(它是 50% 的大头,但也最容易越界撞原则 ①)。
2. **交互反馈基础套**:选项按压 / 发光、**数值滚动渐变**(80→65 慢慢掉,而非瞬跳)、规则点亮仪式感升级。
3. **三个记忆点**:
   - 规则怪谈**首进灯闪一下**;
   - 修仙**境界跳变整页照亮一瞬**;
   - **san < 20 时 UI 轻微错位与文字异常**。
   **三者数据链路全部已有**(`bands` 档位、数值跳变、`san` 绝对值均在现有下发投影里)→ **零后端改动**。
4. **关键时刻两处**:起局揭幕、结局收束的**页面级转场**。

### 7. 显式不在第一版

| 项 | 挂起理由 |
|---|---|
| 物件级动效(丹药飞入等) | 依赖 `stateUpdate` 尚无「获得物品」结构化事件,属 schema 扩展另议 |
| 背景图随世界演变 | 依赖主题 A 持久世界记忆(冻结中),见 [backlog §7](../phase2-polish-and-vision-backlog.md) |
| Particles / Lottie / 轻量 Shader | 缓引(理由见 §3) |
| 声音 | 以后 |

### 8. 执行形态(两段分开)

- **探索段 = Cowork**:开**独立视觉样板间**试效果,**不直接改 `web/src`**。探索要的是快与试错量。
- **落地段 = Claude Code 流程**:feature 分支 + 测试 + Felix 冒烟,守既有工程纪律(atomic commit、lint/build 全绿、`schemaVersion` 不动)。

**两段刻意不混**——探索的快与落地的稳是两种要求,混在一起会同时失去两者。依赖(`package.json`)的实际改动属**落地批**,不在探索段发生。

### 9. 长期约束(护栏三条,常驻生效)

> **补于 2026-07-23**(本 ADR 修订自身,随 Cowork A 批「结构与设计语言」验收通过、B 批开工前落地)。

**为什么要立成常驻约束**:护栏若只写在单轮任务指令里,**下一轮新需求一来即失效**——修仙再加剑影、规则怪谈再加故障、克苏鲁再加扭曲、末日再加沙尘爆闪,每一步都局部合理,合起来就是四个世界一起变成霓虹灯(撞原则 ①③④,也就是本 ADR 反面清单里的「AI PPT 审美」从后门溜回来)。故三条护栏**不写在任务里,写在档案里**。

**① 每个世界只有一个一级记忆点(唯一,不得增设)**

| 世界 | 一级记忆点 |
|---|---|
| 修仙 `cultivation` | **钟鸣共振** bell resonance |
| 规则怪谈 `rules_creepy` | **灯闪** light flicker |
| 克苏鲁 `cthulhu` | **文字异常 / 不确定** uncertain text anomaly |
| 末日 `apocalypse` | **电台信号** radio transmission |

一级 = 该世界的**签名动作**。想让某世界「再多一个记忆点」时,**答案是加强现有那个,不是并列第二个**——并列即稀释,四个世界各自多加一点,最后四个世界长得一样(原则 ③ 失守)。融合世界不新立一级记忆点:**host 记忆点打底 + 渗漏元素点缀**(同 §5 表末行与 [ADR-012](ADR-012-hybrid-axis-merge-strategy.md))。

与 §6.3 第一版三点的关系:规则怪谈灯闪 / 克苏鲁文字异常逐条对应;**修仙**的一级记忆点收敛为**钟鸣共振**(§6.3 的「境界跳变整页照亮」是它的一次表现,不另立一级);**末日电台**为 B 批新增。

**② 动效预算:每屏同时最多 1 个持续环境 + 1 个低频偶发 + 1 个用户触发**

- 持续环境 = 长期在跑的(呼吸感 / 云雾 / 沙尘);
- 低频偶发 = 隔一阵来一下的(灯闪 / 远光扫过 / 电台杂讯);
- 用户触发 = 玩家动作的回应(按压 / 数值滚动 / 规则点亮)。

**预算是硬的**:想加第二个持续环境,先关掉现有那个。这条把原则 ① 从「靠自觉」变成**可数**——冒烟时数得出来,不必争论「算不算影响阅读」。

**③ 新增动效必须声明它替代谁或降级谁,不得只做加法**

任何新效果的提案与实现说明里必须有一句「**它替代 X**」或「**它把 X 降级为 Y**」。加法要付账,这句就是账;没有这句的效果不进落地批。

**三条同时以 [`AGENTS.md`](../../AGENTS.md) 的 Motion Constraints 一节对 Agent 生效**(lockstep 常驻副本,另含正文禁区 / 生命周期清理与 reduced-motion / 性能 / GSAP Skill 通用建议的不适用清单等执行细则)。**两处不一致时以本 ADR 为准**并回改 AGENTS.md。分工:**GSAP Skill 管技术正确性,AGENTS.md 与本 ADR 管审美边界与架构约束。**

### 关键理由

1. **约束的理由消失了,约束就该重估**——ADR-003 §5 的唯一论据是 Taro 迁移安全,而小程序线已随路线 B 冻结;继续遵守等于让一条失效理由锁住当前主线目标。这与 ADR-002→ADR-015 的演进同构,是**预案内的第二例**。
2. **放开手段的同时收紧标准**,才不会滑向方案 B 的失控:白名单让债可数,视觉宪法让「好看」可援引,优先级让资源配比不靠感觉。
3. **两条纪律正交**(eslint 硬线只管平台 IO、不管 npm 依赖),故本次修订**不削弱** ADR-003 真正承重的部分——接口纪律一字不动。
4. **第一版三个记忆点零后端改动**,风险被限在展示层一处;做砸了只回滚前端。

## 已知代价

1. **移植债(核心代价)**:三个库均不可 Taro 复用,回国解冻小程序线时视觉层需重做克制版。缓解:债可数、只在展示层、已挂钩 ①② 解冻评估(§4)。
2. **包体与低端机性能**:GSAP + Motion 会显著增加前端包体(现 dist 844KB 量级),动效越重越挑设备;微信内置 WebView 与低端 Android 是真实风险面。缓解:落地批须实测首屏与动效帧率,`prefers-reduced-motion` 继续全局尊重,不达标就降级或退回 CSS。
3. **视觉宪法靠人守,无自动检查**:不像 eslint 硬线可 CI 拦。缓解:原则 ① 是可判定的(冒烟一眼可验「动画有没有妨碍读字」),其余三条走 Felix 冒烟拍板;每个效果落地时须能回答「服务哪个原则」。
4. **白名单流程摩擦**:想用第四个库要回窗口。接受——摩擦即防膨胀机制。
5. **方案 A/B 的代价**:A 是用失效理由锁死主线目标;B 是无预算无标准的放开(包体/债/审美三头失控)。C 用「名单 + 宪法」换「手段够用且债可数」。

## 重新审视的触发条件

- **回国 / 小程序线解冻**:届时按 §4 显式清偿——评估小程序端视觉层克制版重做成本,可能连带重估白名单(某库若届时已有 Taro 可行方案则留、否则该效果降级)。
- **缓引清单需求出现**(第一版做完发现 CSS+GSAP 的粒子感确实不够 / 动画素材已产出):**回 Project 窗口对齐后**方可转正,不得在实现批自行引入。
- **性能实测不达标**(低端机或微信 WebView 掉帧、首屏明显变慢):降级该效果或退回 CSS;若系统性不达标,重估白名单本身。
- **原则 ① 被违反**:任何被判定「妨碍阅读」的效果**立即回退**,不进入讨论流程——这是宪法里唯一的一票否决项。
- **视觉工作外溢出展示层**:若某效果要求逻辑/状态层改动或新后端字段,停下评估(可能属 schema 扩展或主题 A,另立单元)。
