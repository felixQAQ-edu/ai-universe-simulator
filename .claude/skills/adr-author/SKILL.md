# 撰写 ADR(架构决策记录)

为 Felix 的「AI 后端架构师一年计划」项目撰写一份新的 ADR。本项目的 ADR 有固定模板和工程纪律,严格遵循。

## 现有 ADR(用于定编号与交叉引用)

```!
ls -1 docs/adr/ 2>/dev/null || echo "未找到 docs/adr/,请在仓库根目录调用,或先确认 ADR 存放路径"
```

## 流程

1. **确定编号**:看上面已有的 `ADR-NNN-*.md`,取最大编号 +1,三位补零(已有到 008 则新建 `ADR-009`)。

2. **收齐要素**:优先从当前对话历史里抽取下列内容,缺什么就一次性列成清单问 Felix,不要逐条挤牙膏:
   - 背景与约束条件(为什么现在做、相关 SPI/前序 ADR、并发/阶段等约束)
   - 候选方案(至少 2 个,每个写清优点 / 缺点)
   - 采纳哪个方案 + 关键理由
   - 已知代价(被接受的权衡,能写缓解方式就写)
   - 重新审视的触发条件(尽量挂到里程碑 M2/M5/M6… 或具体事件)
   - 实施步骤(可含 bash 与验证)
   - 与其他文档的交叉引用(LEARNING-NOTES 笔记 N、其他 ADR、源文件)

3. **生成文件**:严格按同目录 [template.md](template.md) 的小节顺序与标题产出,写到 `docs/adr/ADR-NNN-<英文 kebab slug>.md`。slug 用英文短横线,风格对齐现有文件(如 `chat-memory-store-persistence`、`remove-http-client-config`)。

4. **收尾(主动提醒并代办)**:
   - 在 `README.md` 的「技术决策记录(ADR)」列表追加一行链接。
   - 若本决策源自某条 LEARNING-NOTES 笔记,提醒在该笔记末尾加一行「YYYY-MM-DD · 本主题已由 ADR-NNN 处理」,形成"决策 ↔ 经验"双向闭环。

## 本项目约定(务必遵循)

- **标题格式**:`# ADR-NNN · <中文一句话主题>`。
- **元信息三行**:`日期`(默认今天)、`状态`(默认「已采纳」;尚未落地的写「提议」,被替换的写「被 ADR-XXX 取代」)、`决策者`(Felix)。
- 在「候选方案」中,被采纳的方案标题后标注「(本 ADR 采纳)」;被排除的方案要写清**为什么排除**。
- **「已知代价」是本项目 ADR 的灵魂**——诚实列出被接受的权衡,不要美化,能写「缓解方式:…」就写。
- 「重新审视的触发条件」让决策可被未来推翻,要给出客观信号(压测数据、版本升级、新场景等)。
- 「实际效果(事后补充)」留占位,用斜体「*<里程碑/场景>时回填:<要验证什么>*」,因为 Felix 习惯事后用真实数据回填。
- **工程纪律口吻**:一次决策只动一个变量;推翻要有反证证据;反证完不仅要得结论、还要彻底清理实施;删代码保留 git 历史兜底。

完整骨架见 [template.md](template.md),照抄其结构再逐节填内容。
